# 子女端：紧急联系人接口（`emergency_contacts`）

本文档仅描述**紧急联系人**相关接口，字段语义与 `sql/table_v7.sql` 中 `emergency_contacts` 表一致：`priority` 为 **数字越小越优先**（最先联络）。**不使用 `isPrimary`**，联络顺序完全由 `priority` 表达。

**约定：** 基础路径前缀 `/api/v1`；鉴权 `Authorization: Bearer <token>`；路径中 `{elderId}` 为老人档案 id（`elder_profiles.id`）。成功响应建议统一为 `{ "code": 0, "message": "ok", "data": ... }`。

**前端展示：** 列表建议按 `priority` **升序**排列，自上而下为「先联络 → 后联络」。

---

## GET 列表

- **路径：** `GET /api/v1/children/elders/{elderId}/emergency-contacts`
- **查询参数（可选）：** `sort=priority`（按 `priority` 升序）

**响应 `data`：** 对象数组，元素字段：`id`、`elderId`、`name`、`phone`、`relation`、`priority`、`createdAt`、`updatedAt`。

---

## POST 新建（方案 B：唯一约束去重）

- **路径：** `POST /api/v1/children/elders/{elderId}/emergency-contacts`

**请求体：**

```json
{
  "name": "张明",
  "phone": "13900139001",
  "relation": "儿子",
  "priority": 1
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | 姓名 |
| `phone` | 是 | 手机号 |
| `relation` | 是 | 与老人关系 |
| `priority` | 是 | 正整数，**数字越小越优先** |

**去重规则（数据库 V8）：** `sql/table_v8.sql` 为 `emergency_contacts` 增加 `UNIQUE(elder_profile_id, phone)`，保证**同一老人**下手机号不重复（避免重复插入）。  
这属于“去重/防重复数据”，不是严格意义的“幂等重放返回相同成功响应”。重复提交同手机号时，服务端应返回 `409 Conflict`（或业务码表示“已存在”）。

**重复手机号响应示例（建议）：** `409`

```json
{
  "code": 409,
  "message": "该老人下该手机号的紧急联系人已存在",
  "data": null
}
```

## PATCH 部分修改

- **路径：** `PATCH /api/v1/children/elders/{elderId}/emergency-contacts/{contactId}`

**请求体（示例，仅改优先级）：**

```json
{
  "priority": 1
}
```

可只包含需要更新的字段（至少一项）。

- **路径：** `DELETE /api/v1/children/elders/{elderId}/emergency-contacts/{contactId}`
- **请求体：** 无
